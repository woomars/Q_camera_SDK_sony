# 4K 30fps 검증 보고서 (2026-03-31)

## 1. 목표
- 목표 조건: `3840x2160`, `30fps`, 안정적인 캡처 파이프라인
- 사용자 우선순위: `FPS 최우선`

## 2. 검증 방법 (CLI 우선)
- UI 영향을 배제하기 위해 CLI 측정 도구를 구성하고 반복 측정 수행
- 측정 지표:
  - 협상 결과(`해상도/포맷/fps`)
  - 실측 fps(프레임 수 / 실측 시간)
  - 타임스탬프 기반 fps
  - 추정 드롭 프레임 수
  - 페이로드 처리량(MB/s)

## 3. 반영한 코드
- 네이티브 계측/선택 로직: `src/CameraCore/CameraCore.cpp`
  - 협상 정보 조회 API
  - 타임스탬프 fps, 드롭 카운터, 성능 리셋
  - 장치 선택 우선순위/힌트(`CAM_DEVICE_HINT`) 반영
- CLI 측정 도구:
  - `src/CameraCli/CameraCli.csproj`
  - `src/CameraCli/Program.cs`

## 4. 실험 결과

### A. 4K MJPEG (5회, 2초 측정)
- 협상: `3840x2160 @ 30.00 (MJPG)`
- 실측:
  - Run1: `23.88 fps`
  - Run2: `23.92 fps`
  - Run3: `23.88 fps`
  - Run4: `24.00 fps`
  - Run5: `23.98 fps`
- 타임스탬프 fps: 약 `23.8`
- 드롭: `0`

### B. 4K NV12 (5회, 2초 측정)
- 협상: `3840x2160 @ 30.00 (NV12)`
- 실측:
  - Run1: `23.99 fps`
  - Run2: `23.99 fps`
  - Run3: `23.98 fps`
  - Run4: `23.97 fps`
  - Run5: `23.97 fps`
- 타임스탬프 fps: 약 `23.8`
- 드롭: `0`

## 5. 해석
- MJPEG/NV12 모두 `30fps로 협상`되지만 실제 유입 간격은 약 `23.8fps`
- `drop=0`이므로 앱에서 프레임을 잃는 것이 아니라, 소스 스트림 자체가 약 24fps로 전달됨
- 결론적으로 현재 장치/드라이버/펌웨어/환경 조합에서는 앱 최적화만으로 `4K 실 30fps` 달성 불가

## 6. 전략별 효과

### 효과가 있었던 항목 (진단/안정성 측면)
1. CLI 우선 검증 체계
- UI 오버헤드 변수 제거, 재현 가능한 실측 확보

2. 타임스탬프 기반 계측
- "협상 30fps"와 "실유입 24fps"를 명확히 분리

3. 드롭 카운터/리셋
- 앱 내부 드롭이 아닌 소스 출력 한계임을 확인

4. 장치 선택 강화
- 내장 카메라 오선택 위험 감소 (`CAM_DEVICE_HINT`, 외장 우선)

### 30fps 달성에는 효과가 없었던 항목
1. 메모리 큐 확대 / 단기 메모리 전용 녹화
- 버퍼 안정성 개선은 있었으나 소스 fps 상승 없음

2. 저지연 플래그/변환 경로 최소화
- 오버헤드 완화 효과는 있으나 최종 fps는 약 24 유지

3. MJPEG 입력/출력 고정
- CPU 부담 감소는 있으나 4K 조건에서 여전히 약 24fps

## 7. 현재 결론
- **4K 실 30fps 미달성 (검증 완료)**
- 현재 재현 가능한 상한: **약 23.8~24.0fps** (`3840x2160`, MJPG/NV12 공통)

## 8. 권장 후속 조치
1. 벤더 검증
- 해당 모델/펌웨어/드라이버 조합에서 UVC 4K30 실출력 보장 여부 확인

2. 호스트 경로 점검
- USB 컨트롤러 경로, 케이블, 허브, 전원 정책 확인

3. 30fps가 즉시 필수라면
- 해상도 하향으로 30fps 확보 또는 벤더 전용 전송/SDK 경로 검토

## 9. 비고
- 간헐적으로 `0x80070005(Access Denied)`가 발생했으며, 장치 점유/권한 상태 영향으로 판단됨
- 단일 캡처 프로세스 유지 및 장치 힌트 지정으로 재현성 향상
